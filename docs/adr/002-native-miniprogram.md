# ADR-002: 微信小程序选用原生框架

## 状态

已采纳

## 背景

PaperLens V1.0 以微信小程序为主力客户端。小程序开发有多种技术选型：

1. **原生框架**：WXML/WXSS/JS
2. **跨端框架**：Taro、uni-app、Remax

## 决策

选用 **微信小程序原生框架**。

## 理由

### 1. 审核友好

- 原生框架无编译层，代码直接运行
- 第三方框架可能引入审核风险（如 Taro 某些版本被拒）
- V1.0 需快速过审，避免非技术因素延期

### 2. 包体积控制

- 原生框架无运行时依赖
- 跨端框架需打包运行时（~100KB+）
- 小程序主包限制 2MB，原生更可控

### 3. 性能最优

- 无抽象层开销
- 直接使用微信最新 API
- 渲染性能最优

### 4. 学习成本低

- 项目规模小（3 个页面）
- 原生语法简单，无需学习框架 API
- 团队熟悉度高

## 替代方案

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| **原生框架** | 审核友好、性能优、包体小 | 仅限微信，不跨端 | ✅ 采纳 |
| Taro | 多端一套代码 | 包体大、审核风险、学习成本 | ❌ V1.0 不需要 |
| uni-app | Vue 语法、多端 | 同上 | ❌ V1.0 不需要 |
| Remax | React 语法 | 社区较小，同上 | ❌ V1.0 不需要 |

## 后果

- **正面**：
  - 审核顺利，快速上线
  - 性能最优，用户体验好
  - 开发调试直接，无编译等待

- **负面**：
  - 如后续需支持其他平台（支付宝、抖音小程序），需重写
  - 无法复用 Web 端代码（Vue）

- **缓解**：
  - 业务逻辑层保持简洁，UI 层独立
  - 如 V2.0 需多端，可渐进迁移至 Taro

- [Taro 审核风险讨论](https://github.com/NervJS/taro/issues/xxx)
